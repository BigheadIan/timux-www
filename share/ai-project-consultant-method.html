AI 專案顧問方法論

如何談需求、切 MVP、交代 AI、穩定交付產品

智造交給 AI，時光留給所愛

開場提醒：這不是 AI 工具課，而是如何把 AI 當作執行團隊管理。
今天的核心命題

不是「把需求丟給 AI」

而是把自己放在產品負責人、客戶翻譯者、AI 指揮者、交付驗收者的位置。

1

聽懂現場

從業務流程與痛點出發，不急著賣系統。

2

翻成產品

把模糊對話整理成角色、流程、資料與 MVP。

3

管理 AI

給上下文、邊界、優先級與驗收方式。

心法 01

先聽現場，不急著賣系統

常見錯誤

一聽需求就列功能

客戶說「想數位化」，我們馬上開始講 ERP、CRM、AI OCR、庫存、報表。

頭頭做法

先問日常怎麼做

誰在做？用什麼表？哪一步最痛？哪裡最常錯？怎樣算改善成功？

這一頁要讓新人知道，顧問價值不是背功能表，而是找到客戶真正卡住的流程。
方法 01

需求探索：四層問題

業務輪廓

公司規模、角色、服務流程、客戶類型、營收模式。

痛點排序

最耗時、最常錯、最影響收入、老闆最常親自處理。

現況工具

紙本、Excel、LINE、既有系統、照片、表單、人工規則。

決策條件

誰拍板、何時要 Demo、預算型態、可否先試點。

互動練習 1

模糊需求 → 訪談問題

客戶說：我們想做一套 AI 管理系統，現在很多事情都很亂，希望你們幫忙數位化。

請交代 AI：

  • 先不要規劃系統。
  • 先整理要問客戶的問題。
  • 問題要分：業務輪廓、流程、痛點、資料來源、決策條件、Demo 切入點。
方法 02

產品切片：先打最痛的一段流程

現場痛點
→
最短 Demo
→
MVP
→
擴張系統

PoC 不是完整系統。PoC 是讓客戶一眼看出：「你真的懂我的痛」。

案例｜吊桿車維修廠

真痛點不是「我要 ERP」

表面需求

完整管理系統

工單、報價、庫存、排程、客戶、KPI、財務報表全部都要。

第一版切入

老闆每天卡住的流程

工單照片 → AI OCR → 人工確認 → 客戶版報價 → 手機審核/定價。

互動練習 2

功能清單 → MVP 切分

客戶想要：工單、報價、庫存、排程、客戶管理、KPI、LINE 通知、AI OCR、財務報表。

請交代 AI：

  • 分成「第一版必做 / 第二版加分 / 先不要做」。
  • 說明切分理由。
  • 指出第一版 Demo 的最短路徑。
方法 03

複用既有模型：新專案不是從零想

先問：它像我們做過的哪個系統？能抽出哪種母型？

Web Admin LIFF 手機操作 LINE 通知 AI 辨識 人工確認 任務 / 審核 報表
案例｜銘將 → 吊桿車

複用的是產品結構，不是照抄畫面

銘將保全

考勤、文件辨識、LINE 入口、管理員 LIFF、人工確認、薪資報表。

吊桿車維修廠

工單、報價、定價、派工、客戶版明細、庫存後續擴張。

相同：管理後台 + 手機高頻操作 + AI 辨識 + 人工確認。不同：業務資料與決策流程。

錯誤示範

只給目標，AI 就會自由發揮

不要這樣交代

幫我做吊桿車維修廠系統，要有工單、報價、庫存、客戶管理、AI 辨識，越完整越好。

  • 沒說誰最常用。
  • 沒說第一版打哪個痛點。
  • 沒說哪些功能先不要做。
  • 沒定義怎麼驗收。
正確示範

把 AI 拉回真實場景

背景：吊桿車維修廠用手寫工單，老闆每天要整理成客戶報價，還要判斷加價與派工，手機使用頻率高。

目標：先規劃 MVP，不做完整 ERP。第一版展示「拍照 → OCR → 人工確認 → 客戶版報價 → 手機審核」。

參考：銘將保全 Web Admin + LIFF + AI 辨識人工確認。

約束：庫存、KPI、法規提醒先列後續。

交付：模組清單、資料表草案、MVP 流程、第一週開發順序。

AI 溝通情境庫

新人最需要練的 8 種交代

需求訪談前

拆業務輪廓、痛點提問、Demo 切入點。

會議紀錄後

整理角色、流程、資料、待確認問題。

規劃 MVP 時

切第一版、第二版、先不要做。

交代開發時

讀記憶/文件，照既有架構，不重造輪子。

修 bug 時

先復現、找根因、提出修法，再改。

部署前

列 build、測試、備份、回滾、線上驗證。

方法 05

介面判斷：以真實使用場景為準

Web Admin

完整管理、複雜設定、大量表格、後台報表。

LINE / LIFF

手機高頻操作：審核、派工、定價、查詢、回報。

高頻操作少一層就是少一層。

案例｜銘將 LINE

「流程很怪」是非常重要的驗收語言

原流程

打卡 → 上下班選擇 → 同一個打卡頁

多一層選擇，使用者會困惑。

修正

打卡 → 直接進上下班打卡頁

頁面內再選上班/下班，減少入口層級。

方法 06

驗收與糾錯：不要只說「錯了」

好的驗收回饋格式：

現象
→
判斷依據
→
期望流程
→
驗收方式

AI/合作伙伴要回到根因，不要只修表面。

失誤案例庫

失誤不是丟臉，是 SOP 的原料

LIFF 誤判

direct LIFF 不是目標；穩定取得 LINE uid 才是目標。

API 404 誤列

要確認實際頁面有沒有用，不要把參考資料當需求。

部署路徑

容器 running 不代表成功；dist 路徑錯仍會 502。

改圖改到流程

改動要守範圍，不要順手重構或回退別的功能。

SOP 01

大架構先完成

先定義

  • 產品目標
  • 使用角色
  • 核心流程
  • 系統邊界

再落地

  • 技術架構
  • 資料模型
  • MVP 範圍
  • 驗收標準

先不要寫代碼；先標出高風險與需要客戶確認的假設。

SOP 02

建立 repo 與開發基線

  • 建立 GitHub repo 與乾淨 skeleton commit。
  • README 寫清楚啟動方式與架構。
  • 建立 `.env.example`，不得提交 secret。
  • 設定 `.gitignore`，避免 build 產物與金鑰進 repo。
  • 若套用母型，標清楚「複用什麼、不複用什麼」。
SOP 03

本地開發順序：DB → API → 前端


DB Schema
→ prisma validate / migration or db push
→ API module / service / DTO
→ backend build
→ curl 或 API 測試
→ frontend page / integration
→ frontend build
→ browser/mobile 實測
→ 整合驗證
SOP 04

部署前檢查

上線前

  • Git 狀態乾淨
  • 本地 build 通過
  • API curl 測過
  • 重要頁面實測

風險控管

  • 環境變數確認
  • secret 不進聊天
  • 備份或回滾方案
  • 知道上一個可用版本
SOP 05

部署後驗證，不只看服務 running

1
檢查 service / container / PM2 狀態。
2
curl 核心 API，確認 200 與資料格式。
3
瀏覽器檢查核心頁面與新 asset。
4
手機/LIFF/LINE 入口走一遍真實流程。
5
記錄 commit、部署結果、風險與後續事項。
互動練習 3

AI 說「做完了，可以部署」時怎麼追問？

  • build 是否通過？
  • API 是否 curl 測過？
  • 前端是否用瀏覽器測過？
  • 手機/LIFF 是否測過？
  • 有沒有回滾方案？
  • commit 是哪一個？

新人檢核表

你能做到這 6 件事，就能開始跟專案

  1. 把客戶一句話整理成訪談問題。
  2. 把功能清單切成 MVP / 後續 / 暫不做。
  3. 找出可複用的既有專案母型。
  4. 用完整公式交代 AI，而不是只丟目標。
  5. 用真實使用情境驗收畫面與流程。
  6. 照 SOP 完成本地測試、部署與記憶化。
