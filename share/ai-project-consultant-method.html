心法 01

先听业务现场，不急着卖系统

常见误区

一听需求就列功能

客户说“想数字化”，马上开始讲 ERP、CRM、AI OCR（图片文字识别）、库存、报表。

正确做法

先问日常怎么跑

谁在做？用什么表？哪一步最痛？哪里最容易出错？怎么算有效果？

方法 01

需求探索：四层问题

业务轮廓

公司规模、岗位角色、服务流程、客户类型、收入模式。

痛点排序

最耗时、最容易错、最影响收款、老板最常亲自处理。

现有工具

纸质单据、Excel、微信/企微、系统、照片、表单、人工规则。

决策条件

谁拍板、什么时候看 Demo、预算怎么定、能不能先试点。

解释图｜访谈问题地图

参考吊杆车访谈稿：问题要沿着业务链路问

🏢

业务轮廓

规模、客户、岗位、月单量、竞争优势。

🔧

作业流程

从接单、诊断、维修、验收到交付，每一步谁负责。

📍

进度追踪

状态怎么同步？客户怎么问？卡在哪一环？

📦

库存/数据

有没有账实不符、缺料、重复录入、资料散落。

💰

报价/收款

谁报价？怎么对账？哪里会漏单、低报、催款。

重点：不是背问题清单，而是顺着“真实业务链路”一步步找痛点。

方法 02

产品切片：先打最痛的一段流程

第一版 MVP

只做最能让客户“哇”的闭环

① 真实痛点

最耗时、最常错、老板最痛。

② 数据来源

纸质单据、照片、Excel、系统导出。

③ 最短 Demo

一条能跑通、能现场演示的路径。

④ 后续扩展

库存、报表、KPI 等放第二阶段。

解释图｜MVP 取舍矩阵

不是功能越多越好，而是第一版越准越好

第一版必做

高痛点 × 高可展示

能在 Demo 里跑通，客户马上懂价值。

例：拍照工单 → AI 识别 → 客户版报价。

第二版增强

有价值 × 但链路较长

等第一版稳定后，再补看板、库存、报表。

例：低库存预警、自动采购草稿。

先不做

听起来完整 × 但不影响 Demo

先记录，不要拖慢第一版上线。

例：复杂 KPI、全面条码体系。

商业判断 01｜优先做增收场景

客户最愿意买单的，是能帮他赚更多钱的服务

支持系统

ERP / 入档 / 内部管理

能降本、提效、规范流程，但传统客户仍可手动操作，短期收益感不一定强。

增长系统

AI 客服 / 获客 / 转化

能接住更多咨询、提高成交率、扩大业务承接能力，客户更容易感受到“用了就能多赚钱”。

选题优先级：先找客户的收入入口，再补客户的支持系统。

解释图｜价值优先级

从“能不能做”升级为“值不值得先做”

优先切入

直接创造收入

获客、咨询转化、报价成交、复购运营。

例：AI 客服承接咨询，帮需要接业务的公司扩大销售半径。

第二优先

缩短收款链路

报价、合同、请款、对账，让钱更快回来。

例：维修厂工单翻译成报价单，降低漏单少收。

支持能力

内部管理提效

ERP、库存、档案、报表，用于长期稳定运营。

例：AI 识别入档、库存盘点、经营看板。

案例｜吊杆车维修厂

真正痛点不是“我要一套 ERP”

表面需求

完整管理系统

工单、报价、库存、排班、客户、KPI、财务报表都想要。

第一版切入

老板每天卡住的流程

工单照片 → AI OCR（图片文字识别）（图片文字识别） → 人工确认 → 客户版报价 → 手机审批/定价。

  • 说明切分理由。
  • 指出第一版 Demo 的最短路径。

正确示范

把 AI 拉回真实业务场景

背景：维修厂现在用手写工单，老板每天要整理成客户报价，还要判断加价和派工，手机使用频率高。

目标：先规划 MVP（第一版最小可行方案），不做完整 ERP（企业内部管理系统）。第一版展示“拍照 → OCR（图片文字识别） → 人工确认 → 客户版报价 → 手机审批”。

参考：铭将安保的后台 + 手机操作 + AI 识别人工确认流程。

约束：库存、KPI、法规提醒先放到后续阶段。

交付：模块清单、数据表草案、MVP 流程（第一版流程）、第一周开发顺序。

方法 05

界面判断：以真实使用场景为准

Web 管理后台

完整管理、复杂配置、大量表格、经营报表。

手机端入口

高频操作：审批、派工、定价、查询、回报。

高频流程少一层，就是更好用一分。

案例｜手机入口

“流程很别扭”就是重要验收信号

原流程

打卡 → 上下班选择 → 同一个打卡页

入口多一层，用户会困惑。

修正

打卡 → 直接进入打卡页

页面内再选上班/下班，减少操作层级。

复盘案例库

失误不是丢脸，是 SOP 的原料

手机入口误判

直达入口不是目标；稳定识别用户身份才是目标。

API（系统接口）误判

要确认实际页面是否调用，不要把参考资料当需求。

发布路径

服务 running（看似启动）不等于成功；构建产物路径错仍会 502（网站打不开）。

越界修改

改动要守范围，不要顺手重构或回退其他功能。

商业判断 02｜品牌阶段收费策略

不能免费，但要让客户很难拒绝

1

先打品牌

现阶段重点是提升知名度、积累案例、锻炼交付能力。

2

价格有吸引力

报价要让客户觉得“现在不试很可惜”，但不能免费消耗专业价值。

3

订阅降低门槛

用月费/年费/轻量启动包降低初次决策压力，后续随价值扩展。

早期目标不是一次收大钱，而是用可负担的订阅，把客户带进长期合作关系。

术语小抄｜给非工程背景同事

听到这些词，先用白话理解就好

MVP

第一版最小可行方案：不是完整版，而是先做最能证明价值的一小段。

PoC / Demo

概念验证/演示：拿来让客户“看懂我们能解决什么”，不等于正式系统。

ERP

企业内部管理系统：管库存、流程、报表、财务等，偏支持系统。

AI 客服

帮客户接咨询、回答问题、收集线索、提高转化，偏增收系统。

术语小抄｜开发流程翻译

不用会写代码，也要听懂项目进度

代码仓库 / GitHub repo

项目的线上文件夹，所有代码和版本记录都放这里。

数据库 / DB

系统的资料柜：客户、订单、工单、库存等资料都存在这里。

接口 / API

前端页面和后台资料沟通的门，例如“查询订单”“新增客户”。

前端 / 后端

前端是用户看到的页面；后端是看不见但负责计算、存资料的系统。

术语小抄｜发布验收翻译

这些词代表“有没有真的做好”

build / 构建

把代码打包成能运行的版本；如果 build 失败，通常不能上线。

commit

一次代码存档点；出事时能知道改了什么，也能退回上一版。

deploy / 发布上线

把本地做好的版本放到客户能打开的网址或服务器上。

rollback / 回滚

上线出问题时，先退回上一版稳定版本，避免客户继续受影响。

SOP 01

产品架构先完成

先定义

  • 产品目标
  • 使用角色
  • 核心流程
  • 系统边界

再落地

  • 技术架构
  • 数据模型
  • MVP 范围
  • 验收标准
SOP 02

代码仓库与开发基线（项目文件夹和起点）

  • 建立 GitHub 仓库（线上代码文件夹）和干净的 skeleton commit（初始存档点）。
  • README 写清楚启动方式和项目结构。
  • 建立 `.env.example`，不得提交密钥。
  • 设置 `.gitignore`，避免构建产物进入仓库。
  • 如果套用项目母版，写清楚“复用什么、不复用什么”。

SOP 04

发布前后都要验收

上线前

  • Git 状态干净
  • 本地 build（打包检查）通过
  • 接口测试通过
  • 关键页面实测

上线后

  • 服务状态正常
  • 核心 API（系统接口）200
  • 浏览器打开新版
  • 手机端入口跑通
现场练习 A｜美伽职业学院

请把“招生与教学管理”拆成访谈问题

情境设定

美伽职业学院想导入 AI，提高招生转化、学员服务和教务管理效率。校方说“想做 AI 咨询/课程/管理系统”，但具体痛点还不清楚。

练习任务：请交代 AI 产出一份 6 分钟可用的访谈提纲。

必须覆盖：招生来源、咨询话术、报名转化、课程排班、学员跟进、就业/证书、老师工作量。

输出格式：按“必问 / 建议 / 补充”分层，并标出可能的 Demo 切入点。

现场练习 B｜谷品生活 Gooduct Lab

请把“生活品牌运营”拆成 AI 项目机会

情境设定

谷品生活 Gooduct Lab 做生活方式/选品/内容与社群运营，希望用 AI 提高内容产出、选品决策、客户互动和复购运营效率。

练习任务：请交代 AI 先做需求探索，不要直接设计系统。

必须覆盖：选品流程、内容生产、渠道分发、用户画像、订单/库存、社群互动、复购与会员。

输出格式：痛点排序 + MVP 切入建议 + 一条能现场演示的 Demo 路径。

新人检核表

能做到这 8 件事，就能开始跟项目

  1. 把客户一句话整理成访谈问题。
  2. 把功能清单切成 MVP（第一版）/ 后续增强 / 暂不做。
  3. 找出可复用的项目母版。
  4. 判断这个项目优先是“增收”还是“支持”。
  5. 设计低门槛但不免费的订阅切入方案。
  6. 用完整公式交代 AI，而不是只丢目标。
  7. 用真实使用场景验收页面和流程。
  8. 按 SOP 完成本地测试、发布上线和复盘沉淀。
结语

AI 项目顾问的价值

不是比 AI 更会写代码，而是比 AI 更懂现场、更懂收益、更懂取舍、更懂验收。
