智造交给 AI，时光留给所爱

本版改为简体中文、内地表达和更专业的商务培训视觉。
核心定位

不是“把需求丢给 AI”

而是把自己放在产品负责人、客户翻译者、AI 指挥者、交付验收者的位置。

1

听懂现场

从业务流程和真实痛点出发，不急着卖系统。

2

翻成产品

把模糊表达整理成角色、流程、数据和 MVP。

3

管理 AI

给上下文、边界、优先级和验收标准。

心法 01

先听业务现场，不急着卖系统

常见误区

一听需求就列功能

客户说“想数字化”，马上开始讲 ERP、CRM、AI OCR、库存、报表。

正确做法

先问日常怎么跑

谁在做？用什么表？哪一步最痛？哪里最容易出错？怎么算有效果？

方法 01

需求探索：四层问题

业务轮廓

公司规模、岗位角色、服务流程、客户类型、收入模式。

痛点排序

最耗时、最容易错、最影响收款、老板最常亲自处理。

现有工具

纸质单据、Excel、微信/企微、系统、照片、表单、人工规则。

决策条件

谁拍板、什么时候看 Demo、预算怎么定、能不能先试点。

解释图｜访谈问题地图

参考吊杆车访谈稿：问题要沿着业务链路问

🏢

业务轮廓

规模、客户、岗位、月单量、竞争优势。

🔧

作业流程

从接单、诊断、维修、验收到交付，每一步谁负责。

📍

进度追踪

状态怎么同步？客户怎么问？卡在哪一环？

📦

库存/数据

有没有账实不符、缺料、重复录入、资料散落。

💰

报价/收款

谁报价？怎么对账？哪里会漏单、低报、催款。

重点：不是背问题清单，而是顺着“真实业务链路”一步步找痛点。

课堂互动 1

模糊需求 → 访谈问题

客户说：我们想做一套 AI 管理系统，现在很多事情都很乱，希望你们帮忙数字化。

请交代 AI：

  • 先不要规划系统。
  • 先整理要问客户的问题。
  • 问题要分：业务轮廓、流程、痛点、数据来源、决策条件、Demo 切入点。
方法 02

产品切片：先打最痛的一段流程

第一版 MVP

只做最能让客户“哇”的闭环

① 真实痛点

最耗时、最常错、老板最痛。

② 数据来源

纸质单据、照片、Excel、系统导出。

③ 最短 Demo

一条能跑通、能现场演示的路径。

④ 后续扩展

库存、报表、KPI 等放第二阶段。

解释图｜MVP 取舍矩阵

不是功能越多越好，而是第一版越准越好

第一版必做

高痛点 × 高可展示

能在 Demo 里跑通，客户马上懂价值。

例：拍照工单 → AI 识别 → 客户版报价。

第二版增强

有价值 × 但链路较长

等第一版稳定后，再补看板、库存、报表。

例：低库存预警、自动采购草稿。

先不做

听起来完整 × 但不影响 Demo

先记录，不要拖慢第一版上线。

例：复杂 KPI、全面条码体系。

案例｜吊杆车维修厂

真正痛点不是“我要一套 ERP”

表面需求

完整管理系统

工单、报价、库存、排班、客户、KPI、财务报表都想要。

第一版切入

老板每天卡住的流程

工单照片 → AI OCR → 人工确认 → 客户版报价 → 手机审批/定价。

课堂互动 2

功能清单 → MVP 切分

客户想要：工单、报价、库存、排班、客户管理、KPI、消息通知、AI OCR、财务报表。

请交代 AI：

  • 分成“第一版必做 / 第二版增强 / 先不做”。
  • 说明切分理由。
  • 指出第一版 Demo 的最短路径。
方法 03

复用项目母版：新项目不是从零想

先问：它像我们做过的哪个项目？能抽出哪套产品母版？

Web 管理后台手机端高频操作消息通知AI 识别人工确认任务 / 审批经营报表
案例｜铭将安保 → 吊杆车

复用的是产品结构，不是照搬界面

铭将安保

考勤、单据识别、消息入口、管理员手机操作、人工确认、薪资报表。

吊杆车维修厂

工单、报价、定价、派工、客户版明细、后续库存扩展。

相同：后台 + 手机高频操作 + AI 识别 + 人工确认。不同：业务数据和决策流程。

不要这样交代

正确示范

把 AI 拉回真实业务场景

背景：维修厂现在用手写工单，老板每天要整理成客户报价，还要判断加价和派工，手机使用频率高。

目标：先规划 MVP，不做完整 ERP。第一版展示“拍照 → OCR → 人工确认 → 客户版报价 → 手机审批”。

参考：铭将安保的后台 + 手机操作 + AI 识别人工确认流程。

约束：库存、KPI、法规提醒先放到后续阶段。

交付：模块清单、数据表草案、MVP 流程、第一周开发顺序。

AI 沟通场景库

新人最需要练的 6 种交代

访谈前

拆业务轮廓、痛点问题、Demo 切入点。

会议后

整理角色、流程、数据、待确认事项。

规划 MVP

切第一版、第二版、先不做。

交代开发

读项目记忆/技术文档，按现有架构改。

修 Bug

先复现、找根因、提出方案，再修改。

发布前

列测试、备份、回滚、线上验证步骤。

方法 05

界面判断：以真实使用场景为准

Web 管理后台

完整管理、复杂配置、大量表格、经营报表。

手机端入口

高频操作：审批、派工、定价、查询、回报。

高频流程少一层，就是更好用一分。

案例｜手机入口

“流程很别扭”就是重要验收信号

原流程

打卡 → 上下班选择 → 同一个打卡页

入口多一层，用户会困惑。

修正

打卡 → 直接进入打卡页

页面内再选上班/下班，减少操作层级。

复盘案例库

失误不是丢脸，是 SOP 的原料

手机入口误判

直达入口不是目标；稳定识别用户身份才是目标。

API 误判

要确认实际页面是否调用，不要把参考资料当需求。

发布路径

服务 running 不等于成功；构建产物路径错仍会 502。

越界修改

改动要守范围，不要顺手重构或回退其他功能。

SOP 01

产品架构先完成

先定义

  • 产品目标
  • 使用角色
  • 核心流程
  • 系统边界

再落地

  • 技术架构
  • 数据模型
  • MVP 范围
  • 验收标准
SOP 02

代码仓库与开发基线

  • 建立 GitHub 仓库和干净的 skeleton commit。
  • README 写清楚启动方式和项目结构。
  • 建立 `.env.example`，不得提交密钥。
  • 设置 `.gitignore`，避免构建产物进入仓库。
  • 如果套用项目母版，写清楚“复用什么、不复用什么”。
SOP 03

本地开发顺序：数据 → 接口 → 页面

→ prisma validate / migration or db push
→ API module / service / DTO
→ backend build
→ curl 或接口测试
→ frontend page / integration
→ frontend build
→ browser/mobile 实测
→ 集成验证
SOP 04

发布前后都要验收

上线前

  • Git 状态干净
  • 本地 build 通过
  • 接口测试通过
  • 关键页面实测

上线后

  • 服务状态正常
  • 核心 API 200
  • 浏览器打开新版
  • 手机端入口跑通
现场练习 A｜美伽职业学院

请把“招生与教学管理”拆成访谈问题

情境设定

美伽职业学院想导入 AI，提高招生转化、学员服务和教务管理效率。校方说“想做 AI 咨询/课程/管理系统”，但具体痛点还不清楚。

练习任务：请交代 AI 产出一份 6 分钟可用的访谈提纲。

必须覆盖：招生来源、咨询话术、报名转化、课程排班、学员跟进、就业/证书、老师工作量。

输出格式：按“必问 / 建议 / 补充”分层，并标出可能的 Demo 切入点。

现场练习 B｜谷品生活 Gooduct Lab

请把“生活品牌运营”拆成 AI 项目机会

情境设定

谷品生活 Gooduct Lab 做生活方式/选品/内容与社群运营，希望用 AI 提高内容产出、选品决策、客户互动和复购运营效率。

练习任务：请交代 AI 先做需求探索，不要直接设计系统。

必须覆盖：选品流程、内容生产、渠道分发、用户画像、订单/库存、社群互动、复购与会员。

输出格式：痛点排序 + MVP 切入建议 + 一条能现场演示的 Demo 路径。

新人检核表

能做到这 6 件事，就能开始跟项目

  1. 把客户一句话整理成访谈问题。
  2. 把功能清单切成 MVP / 后续增强 / 暂不做。
  3. 找出可复用的项目母版。
  4. 用完整公式交代 AI，而不是只丢目标。
  5. 用真实使用场景验收页面和流程。
  6. 按 SOP 完成本地测试、发布上线和复盘沉淀。
结语

AI 项目顾问的价值

不是比 AI 更会写代码，而是比 AI 更懂现场、更懂取舍、更懂验收。
